# Configuración Fiscal — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a single "Configuración Fiscal" window that consolidates SII, TBAI, and Verifactu configuration into a territory-aware, guided React UI backed by 3 independent NEO entities.

**Architecture:** Three data-only NEO artifacts (`sii-config`, `tbai-config`, `verifactu-config`) expose the existing fiscal config tables as API endpoints. A fourth `fiscal-config` custom window (`layoutType: "custom"`) holds the menu entry and orchestrates a React component that detects the org's fiscal profile and renders only the relevant section(s).

**Tech Stack:** Node.js CLI pipeline (extract-fields.js → resolve-curated.js → push-to-neo.js), React + custom hooks, i18n via useUI() from genericLabels, NEO Headless REST API (`/sws/neo/{spec}/{entity}`), node:test for utils tests.

**Spec:** `docs/superpowers/specs/2026-04-27-fiscal-config-design.md`

Create `tools/app-shell/src/windows/custom/fiscal-config/fiscalConfig.utils.js`:

```js
/**
 * Derives the fiscal profile for an org from the presence/state of its 3 config records.
 *
 * @param {object|null} sii        - SII config record or null
 * @param {object|null} tbai       - TBAI config record or null
 * @param {object|null} verifactu  - Verifactu config record or null
 * @returns {'unconfigured'|'sii'|'sii-navarra'|'sii+tbai'|'tbai'|'verifactu'|'conflict'}
 */
export function detectProfile(sii, tbai, verifactu) {
  if (verifactu && (sii || tbai)) return 'conflict';
  if (verifactu) return 'verifactu';
  if (sii && sii.guipuzcoa === 'Y') return 'sii+tbai';
  if (sii && sii.navarra === 'Y') return 'sii-navarra';
  if (sii) return 'sii';
  if (tbai) return 'tbai';
  return 'unconfigured';
}

/**
 * Returns field defaults for record creation based on territory + SII enrollment.
 * Each property is an object of field overrides, or null if that system should not be created.
 *
 * ⚠️ The etsgSifTerritory values ('ALAVA', 'BIZKAIA', 'GIPUZKOA') are placeholders.
 * Replace them with the actual AD List values from artifacts/tbai-config/schema-raw.json.
 *
 * @param {'espania'|'canarias'|'ceuta'|'navarra'|'alava'|'bizkaia'|'gipuzkoa'} territory
 * @param {boolean} inSii - true if org is enrolled in SII
 * @returns {{ sii: object|null, verifactu: object|null, tbai: object|null }}
 */
export function getTerritoryDefaults(territory, inSii) {
  switch (territory) {
    case 'espania':
      return inSii
        ? { sii: { taxtype: 'IVA' }, verifactu: null, tbai: null }
        : { sii: null, verifactu: { taxtype: 'IVA' }, tbai: null };
    case 'canarias':
      return inSii
        ? { sii: { taxtype: 'IGIC' }, verifactu: null, tbai: null }
        : { sii: null, verifactu: { taxtype: 'IGIC' }, tbai: null };
    case 'ceuta':
      return inSii
        ? { sii: { taxtype: 'IPSI' }, verifactu: null, tbai: null }
        : { sii: null, verifactu: { taxtype: 'IPSI' }, tbai: null };
    case 'navarra':
      return { sii: { navarra: 'Y', taxtype: 'IVA' }, verifactu: null, tbai: null };
    case 'alava':
      return inSii
        ? { sii: { taxtype: 'IVA' }, verifactu: null, tbai: { etsgSifTerritory: 'ALAVA' } }
        : { sii: null, verifactu: null, tbai: { etsgSifTerritory: 'ALAVA' } };
    case 'bizkaia':
      return inSii
        ? { sii: { taxtype: 'IVA' }, verifactu: null, tbai: { etsgSifTerritory: 'BIZKAIA' } }
        : { sii: null, verifactu: null, tbai: { etsgSifTerritory: 'BIZKAIA' } };
    case 'gipuzkoa':
      return inSii
        ? { sii: { guipuzcoa: 'Y', taxtype: 'IVA' }, verifactu: null, tbai: { etsgSifTerritory: 'GIPUZKOA' } }
        : { sii: null, verifactu: null, tbai: { etsgSifTerritory: 'GIPUZKOA' } };
    default:
      return { sii: null, verifactu: null, tbai: null };
  }
}
```

URL convention: The custom window receives `apiBaseUrl` = `{API_BASE}/fiscal-config`. The function `neoBase(apiBaseUrl)` strips the last segment → `{API_BASE}`. Then the 3 data specs are at `{API_BASE}/sii-config/{ENTITY}`, etc.

- [x] **Step 1: Create the hook file**

Create `tools/app-shell/src/windows/custom/fiscal-config/useFiscalConfig.js`:

```js
import { useState, useEffect, useCallback } from 'react';
import { detectProfile } from './fiscalConfig.utils.js';
import { neoBase } from '@/components/related-documents/helpers.js';

// ⚠️ Verify these values from contract.json after running the pipeline (Task 1-3):
//   artifacts/sii-config/contract.json → backendContract.window.primaryEntity
//   artifacts/tbai-config/contract.json → backendContract.window.primaryEntity
//   artifacts/verifactu-config/contract.json → backendContract.window.primaryEntity
const SII_ENTITY = 'siiConfiguration';
const TBAI_ENTITY = 'tbaiConfiguration';
const VERIFACTU_ENTITY = 'verifactuConfiguration';

async function fetchRecord(base, specName, entityName, orgId, token) {
  const params = new URLSearchParams({ AD_Org_ID: orgId, _limit: '1' });
  const res = await fetch(`${base}/${specName}/${entityName}?${params}`, {
    headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' },
  });
  if (!res.ok) return null;
  const json = await res.json();
  return json?.response?.data?.[0] ?? null;
}

export function useFiscalConfig(orgId, token, apiBaseUrl) {
  const [state, setState] = useState({
    loading: false,
    error: null,
    profile: null,
    siiRecord: null,
    tbaiRecord: null,
    verifactuRecord: null,
  });

  const load = useCallback(async () => {
    if (!orgId) {
      setState({ loading: false, error: null, profile: 'unconfigured', siiRecord: null, tbaiRecord: null, verifactuRecord: null });
      return;
    }
    setState(s => ({ ...s, loading: true, error: null }));
    try {
      const base = neoBase(apiBaseUrl);
      const [sii, tbai, verifactu] = await Promise.all([
        fetchRecord(base, 'sii-config', SII_ENTITY, orgId, token),
        fetchRecord(base, 'tbai-config', TBAI_ENTITY, orgId, token),
        fetchRecord(base, 'verifactu-config', VERIFACTU_ENTITY, orgId, token),
      ]);
      setState({
        loading: false,
        error: null,
        siiRecord: sii,
        tbaiRecord: tbai,
        verifactuRecord: verifactu,
        profile: detectProfile(sii, tbai, verifactu),
      });
    } catch (err) {
      setState(s => ({ ...s, loading: false, error: err.message }));
    }
  }, [orgId, token, apiBaseUrl]);

  useEffect(() => { load(); }, [load]);

  return { ...state, refetch: load };
}
```

The original plan had CertModal doing simulated verification (1.4s timeout). This task replaced that with a real backend.

- [x] **Step 1: Add `POST /sws/neo/certificate` handler**

`NeoCertificateHelper.handleCertificateUpload` accepts `multipart/form-data` with `certificate` (file), `orgId`, `password`, optional `setOrgNif`. Delegates to `AddCertificateToOrg.doExecute()` via reflection (same pattern as `NeoProcessService`).

- [x] **Step 2: Add `GET /sws/neo/certificate?orgId=` handler**

`NeoCertificateHelper.handleCertificateGet` queries `etsg_certificate WHERE ad_org_id = :orgId AND isactive = 'Y'` and returns `{exists, validTo}`. Used by `CertSection` on mount to restore state after window refresh.

- [x] **Step 3: NIF/CIF validation**

Before delegating to `AddCertificateToOrg`, the handler parses the X.509 subject DN to extract the org NIF. Priority order:
1. `organizationIdentifier` (OID 2.5.4.97) — FNMT RPJ certs (company CIF).
2. `SERIALNUMBER` (OID 2.5.4.5) — personal/autónomo certs (person's NIF).
3. `CN=… (R: NIF)` — fallback for older FNMT format.

Comparison with `ad_orginfo.taxid` (SAVEPOINT-protected query). `'?'`, empty, and null taxid values are treated as "no NIF configured".

Three outcomes:
- **Match** → proceed.
- **No org NIF** → return `{pendingNifConfirmation: true, certNif}`. On re-POST with `setOrgNif=true`, write NIF to `ad_orginfo.taxid` and proceed.
- **Mismatch** → return HTTP 422.

- [x] **Step 4: Route in NeoServlet**

Updated `NeoServlet` routing to accept both `GET` and `POST` on the `certificate` path.

- [x] **Step 5: Unit tests**

`NeoCertificateHelperTest` (15 tests) covers `parseNifFromDn` for all certificate formats (FNMT RPJ, IDCES prefix, VATID prefix, OID attribute names, CN fallback, null/empty input) and `normalizeNif`.
