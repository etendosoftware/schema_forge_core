# Contract v2 Improvements

> Enhancements to `contract.json` to close the gap between what Schema Forge knows and what reaches the frontend/backend contracts.

**Status:** ALL PHASES COMPLETED

## Problem Statement

The current `contract.json` generated by `generate-contract.js` had three gaps:

1. ~~**Endpoints are hardcoded assumptions.**~~ **DONE** — `generateApiPrediction()` now predicts CRUD, selectors, actions, and query params.
2. ~~**Behavioral metadata is lost.**~~ **DONE** — Callouts, displayLogic, readOnlyLogic flow through the contract. Frontend generator emits TODO comments for callout/onchangefunction translation. See also `hybrid-display-logic.md` for the server-side fallback proposal.
3. ~~**UI hints are missing.**~~ **DONE** — defaultValue, help, fieldGroup, precision, isIdentifier, isSelectionColumn, isFilterable, isTranslated extracted and consumed by frontend generator.

## Implementation Roadmap

| Phase | What | Status | Data Source | Files Affected |
|-------|------|--------|-------------|----------------|
| **1** | Behavioral metadata | **COMPLETED** | `schema-curated.json`, `rules-curated.json` | `generate-contract.js`, `generate-frontend.js` |

## Phase 1: Behavioral Metadata

Data that IS extracted (in `schema-raw.json` and `rules-raw.json`) but does NOT reach `contract.json`. These additions go in `frontendContract` only — the backend does not need them.

### 1.1 Callouts

Java classes triggered on field blur in the classic Etendo UI. Each callout modifies other fields in response to a change.

**Important:** Callouts do NOT execute in the NEO Headless API. This metadata is informational — it tells the frontend team which fields have server-side derivation logic that may need to be replicated or replaced.

**Source:** `schema-curated.json` field entries (callout class name), `rules-curated.json` (trigger column, detected effects, complexity, LOC).

**Proposed field addition:**
```json
{
  "name": "businessPartner",
  "callout": {
    "className": "org.openbravo.erpCommon.ad_callouts.SE_Order_BPartner",
    "effects": ["partnerAddress", "invoiceAddress", "priceList", "paymentTerms", "paymentMethod"],
    "complexity": "high"
  }
}
```

Fields without a callout get `"callout": null`.

### 1.2 Display Logic

Expressions that conditionally show or hide a field based on other field values.

**Source:** `AD_Field.DisplayLogic` column, already present in `schema-curated.json`.

**Proposed field addition:**
```json
{
  "name": "someField",
  "displayLogic": {
    "raw": "@DocStatus@='DR'",
    "js": "docStatus === 'DR'",
    "grid": null
  }
}
```

- `raw` — the original AD expression using `@FieldName@` notation.
- `js` — translated to a JavaScript expression using camelCase field names.
- `grid` — separate display logic for grid view (`AD_Field.DisplayLogicGrid`), or `null` if none.

Fields without display logic get `"displayLogic": null`.

### 1.3 Read-Only Logic

Expressions that make a field conditionally read-only.

**Source:** `AD_Column.ReadOnlyLogic` or `AD_Field.ReadOnlyLogic`, already present in `schema-curated.json`.

**Proposed field addition:**
```json
{
  "name": "someField",
  "readOnlyLogic": {
    "raw": "@Processed@='Y'",
    "js": "processed === 'Y'"
  }
}
```

Fields without read-only logic get `"readOnlyLogic": null`.

### 1.4 Implementation Steps (Phase 1)

1. In `generate-contract.js`, locate `generateFrontendContract()`.
2. For each field, look up callout info from the curated schema/rules.
3. Add `callout`, `displayLogic`, and `readOnlyLogic` keys to each field object.
4. Write a helper `translateAdExpression(raw)` that converts `@FieldName@` to camelCase JS expressions. Handle operators: `=` to `===`, `!` to `!==`, `&` to `&&`, `|` to `||`, `Y`/`N` as string literals.

## Phase 2: Endpoint Prediction

Currently `backendContract.endpoints` are hardcoded assumptions. NEO Headless determines endpoints dynamically from `ETGO_SF_ENTITY` flags and field metadata.

The contract should predict the full API surface.

### 2.1 CRUD Endpoints

Determined by 6 individual boolean flags on `ETGO_SF_ENTITY`:

| Flag | HTTP Method | URL Pattern |
|------|-------------|-------------|
| `IsGet` | `GET` | `/sws/neo/{spec}/{entity}` |
| `IsGetbyid` | `GET` | `/sws/neo/{spec}/{entity}/{id}` |
| `IsPost` | `POST` | `/sws/neo/{spec}/{entity}` |
| `IsPut` | `PUT` | `/sws/neo/{spec}/{entity}/{id}` |
| `IsPatch` | `PATCH` | `/sws/neo/{spec}/{entity}/{id}` |
| `IsDelete` | `DELETE` | `/sws/neo/{spec}/{entity}/{id}` |

Currently the contract assumes all methods are enabled for all entities. The prediction should reflect the actual flags.

### 2.2 Selector Endpoints

Auto-generated for any included FK field with one of these reference types:

- **18** — Table
- **19** — TableDir
- **30** — Search
- **OBUISEL** — UI Selector

Pattern: `GET /sws/neo/{spec}/{entity}/selectors/{columnName}`

Response shape:
```json
{
  "items": [{"id": "...", "label": "...", "...gridFields": "..."}],
  "totalCount": 42,
  "limit": 100,
  "offset": 0,
  "hasMore": false
}
```

The contract should list every selector endpoint derived from included FK fields.

### 2.3 Button Action Endpoints

Auto-generated for Button fields (`AD_Reference_ID = 28`) that have a linked process (`AD_Column.AD_Process_ID`).

| Operation | Pattern |
|-----------|---------|
| List actions | `GET /sws/neo/{spec}/{entity}/{recordId}/action` |
| Execute action | `POST /sws/neo/{spec}/{entity}/{recordId}/action/{columnName}` |

### 2.4 Process Spec Endpoints

For specs with `specType = 'P'` (process-only, no CRUD):

| Operation | Pattern |
|-----------|---------|
| Describe parameters | `GET /sws/neo/{spec}` |
| Execute | `POST /sws/neo/{spec}` |

### 2.5 Discovery Endpoint

Always available: `GET /sws/neo/` — lists all accessible specs for the authenticated user.

### 2.6 Query Parameters

All list endpoints support:

| Parameter | Purpose | Example |
|-----------|---------|---------|
| `_startRow`, `_endRow` | Pagination (default 0-100) | `?_startRow=0&_endRow=50` |
| `_sortBy` | Sorting by field name | `?_sortBy=orderDate` |
| `{fieldName}` | Filtering on searchable fields | `?businessPartner=abc` |
| `parentId` | Parent filter for child entities | `?parentId=123` |

### 2.7 Proposed Contract Structure

Add a top-level `apiPrediction` section:

```json
"apiPrediction": {
  "baseUrl": "/sws/neo/{specName}",
  "crud": {
    "order": {
      "get": true,
      "getById": true,
      "post": true,
      "put": true,
      "patch": true,
      "delete": true,
      "listUrl": "/sws/neo/sales-order/order",
      "detailUrl": "/sws/neo/sales-order/order/{id}"
    }
  },
  "selectors": [
    {
      "entity": "order",
      "column": "C_BPartner_ID",
      "field": "businessPartner",
      "url": "/sws/neo/sales-order/order/selectors/businessPartner"
    }
  ],
  "actions": [
    {
      "entity": "order",
      "column": "DocAction",
      "processName": "Complete Order",
      "url": "/sws/neo/sales-order/order/{id}/action/docAction"
    }
  ],
  "queryParams": {
    "pagination": "_startRow, _endRow (default 0-100)",
    "sorting": "_sortBy={fieldName}",
    "filtering": "?{fieldName}={value} for searchable fields",
    "parentFilter": "parentId={id} for child entities"
  }
}
```

### 2.8 Implementation Steps (Phase 2)

1. In `generate-contract.js`, add a new function `generateApiPrediction(schema, entities)`.
2. For each entity, read the 6 boolean flags to populate `crud`.
3. Scan included fields: any with `reference` type in `[18, 19, 30, 'OBUISEL']` gets a selector entry.
4. Scan included fields: any with `AD_Reference_ID = 28` and `AD_Process_ID` gets an action entry.
5. Add the `queryParams` object as static documentation.
6. Attach the result as `apiPrediction` at the top level of `contract.json`.

## Phase 3: UI Hints Extraction

AD metadata useful for frontend rendering. Requires adding columns to the `extract-fields.js` SQL query and extending `buildSchema()` in the extraction pipeline.

### 3.1 High Priority

| Feature | AD Column | Purpose | Contract Key |
|---------|-----------|---------|--------------|
| Default value | `AD_Column.DefaultValue` | Pre-fill fields on create forms | `defaultValue` |
| Identifier field | `AD_Column.IsIdentifier` | Fields shown in FK dropdown labels | `isIdentifier` |

### 3.2 Medium Priority

| Feature | AD Column | Purpose | Contract Key |
|---------|-----------|---------|--------------|
| Help / tooltip | `AD_Field.HelpComment` or `AD_Column.Help` | Tooltip text for the field | `help` |
| Field group | `AD_FieldGroup.Name` via `AD_Field.AD_FieldGroup_ID` | UI section grouping | `fieldGroup` |
| Selection column | `AD_Column.IsSelectionColumn` | Appears in FK selector grid | `isSelectionColumn` |
| Filterable | `AD_Column.IsFilterable` | Can be used as a query filter | `isFilterable` |
| Precision | `AD_Column.Precision` | Decimal places for amount fields | `precision` |
| Translated | `AD_Column.IsTranslated` | Multi-language values | `isTranslated` |

### 3.3 Low Priority (Future)

| Feature | AD Column | Purpose | Contract Key |
|---------|-----------|---------|--------------|
| Button process | `AD_Column.AD_Process_ID` | Field-level button processes | `linkedProcess` |
| Allow logging | `AD_Column.IsAllowLogging` | Audit trail flag | `isLoggable` |

### 3.4 Implementation Steps (Phase 3)

1. In `extract-fields.js`, add the missing columns to the SQL `SELECT` statement. Join `AD_FieldGroup` for the group name.
2. In the `buildSchema()` function, map the new columns to schema field properties.
3. In `generate-contract.js`, extend `generateFrontendContract()` to include the new keys.
4. Fields without a value for optional hints get `null` (not omitted), so the contract shape is always consistent.

## Phase 4: Contract Tests

Verification that the new contract sections are correct and complete.

### 4.1 New Test Cases

Add to `run-contract-tests.js`:

| Test | Assertion |
|------|-----------|
| Callout fields exist | Every field with `callout !== null` must reference a class name that exists in `rules-curated.json` |
| Display logic is valid JS | Every `displayLogic.js` expression must parse without syntax errors |
| Read-only logic is valid JS | Every `readOnlyLogic.js` expression must parse without syntax errors |
| Selector endpoints match FK fields | Every included FK field (ref 18/19/30/OBUISEL) must have a corresponding entry in `apiPrediction.selectors` |
| Action endpoints match button fields | Every included button field (ref 28) with a process must have an entry in `apiPrediction.actions` |
| CRUD flags are booleans | All 6 flags in each `apiPrediction.crud` entry must be boolean |
| Default values are strings or null | `defaultValue` must be `string | null` |

### 4.2 Implementation Steps (Phase 4)

1. Add a new test group `"Contract v2 checks"` in `run-contract-tests.js`.
2. For JS expression validation, use `new Function(expr)` to check parse-ability (no eval).
3. Optionally extend `generate-contract.js` to emit a `testManifest` section listing all expected selectors and actions for easier test cross-referencing.

### Proposed (All Phases Complete)

{
  "name": "businessPartner",

  "type": "foreignKey",
  "tsType": "string",
  "visibility": "editable",

  "inputMode": "search",

    "className": "org.openbravo.erpCommon.ad_callouts.SE_Order_BPartner",

  "readOnlyLogic": null,

  "defaultValue": null,
  "isIdentifier": true,

- Phase 3 adds `defaultValue`, `isIdentifier`, `help`, `fieldGroup`, `isSelectionColumn`, `isFilterable`, `precision`, `isTranslated` (UI hints).
