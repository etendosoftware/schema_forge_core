# Plan: NEO Headless Report Endpoint

**Status:** Pending
**Date:** 2026-03-12
**Related:** [Process & Report Pipeline](process-and-report-pipeline.md) — Phase 2

## Goal

Enable report execution via NEO Headless (`/sws/neo/`) by reusing the existing `ReportingUtils` from Etendo core. This avoids duplicating Jasper logic and provides a JWT-authenticated, OpenAPI-documented report endpoint.

**Key constraint:** No API changes in `modules_core`. No removal or modification of existing utilities. Only additive changes in `com.etendoerp.go`.

---

## Architecture

```
Frontend (Schema Forge generated)
    │
    │  POST /sws/neo/{specName}/generate
    │  Body: { "exportType": "pdf", "params": { "dateFrom": "..." } }
    │  Auth: JWT Bearer token
    │
    ▼
NeoServlet (existing, path routing)
    │
    ▼
NeoReportService (NEW)
    │  1. Resolve report template path from SFSpec → AD_Process.JasperReport
    │  2. Load report parameters from request
    │  3. Call ReportingUtils.exportJR() — EXISTING static utility
    │
    ▼
ReportingUtils.exportJR() (EXISTING, modules_core, untouched)
    │  Compiles .jrxml → fills with data → exports to format
    │
    ▼
Binary response (PDF/XLS/HTML/CSV)
    Content-Type: application/pdf
    Content-Disposition: attachment; filename="report.pdf"
```

## Implementation Phases

### Phase A: NeoReportService (~100 lines, new file)

**File:** `modules/com.etendoerp.go/src/com/etendoerp/neo/NeoReportService.java`

**Responsibilities:**
1. Resolve the `.jrxml` template path from `AD_Process.JasperReport`
2. Map request parameters to Jasper parameter map (`Map<String, Object>`)
3. Determine export type from request (`pdf`, `xls`, `xlsx`, `html`, `csv`)
4. Call `ReportingUtils.exportJR(templatePath, exportType, params, outputStream)`
5. Return the generated bytes + content type + suggested filename

**Key method:**
```java
public class NeoReportService {

  public static ReportResult generateReport(
      String processId,
      String exportType,
      Map<String, String> params
  ) throws OBException {
    // 1. Look up AD_Process to get JasperReport path
    Process process = OBDal.getInstance().get(Process.class, processId);
    if (process == null || !process.isReport()) {
      throw new OBException("Process is not a report: " + processId);
    }
    String jrTemplatePath = resolveTemplatePath(process.getJasperReport());

    // 2. Build Jasper parameter map
    Map<String, Object> jrParams = buildJasperParams(process, params);

    // 3. Export to byte array via ReportingUtils
    ByteArrayOutputStream baos = new ByteArrayOutputStream();
    ExportType expType = ReportingUtils.getExportType(exportType);
    ReportingUtils.exportJR(jrTemplatePath, expType, jrParams, baos, false, null, null, null);

    // 4. Return result
    return new ReportResult(
      baos.toByteArray(),
      expType.getContentType(),
      buildFilename(process.getName(), exportType)
    );
  }
}
```

**Parameter loading** — Adapt from `BaseReportActionHandler.loadFilterParams()` (copy logic, ~50 lines). This loads `AD_Process_Para` definitions and maps string values to correct Java types (Date, BigDecimal, etc.).

**Template resolution** — `AD_Process.JasperReport` stores a relative path like `org.openbravo.erpCommon.ad_reports/GeneralLedgerReport.jrxml`. Resolve against Etendo's base path + modules path (same logic as `BaseReportActionHandler.getReportTemplatePath()`).

### Phase B: Binary Response Support in NeoServlet (~30 lines, modify existing)

**File:** `modules/com.etendoerp.go/src/com/etendoerp/neo/NeoServlet.java`

**Current state:** `writeResponse()` only writes JSON via `response.getWriter()`.

**Changes needed:**

1. Add a `writeBinaryResponse()` method:
```java
private void writeBinaryResponse(
    HttpServletResponse response,
    byte[] data,
    String contentType,
    String filename
) throws IOException {
  response.setContentType(contentType);
  response.setContentLength(data.length);
  response.setHeader("Content-Disposition",
      "attachment; filename=\"" + filename + "\"");
  response.getOutputStream().write(data);
  response.getOutputStream().flush();
}
```

2. In the `doPost()` routing, add a branch for report specs:
```java
// After existing process handling
if (isReportSpec(spec)) {
  ReportResult result = NeoReportService.generateReport(
      spec.getProcess().getId(),
      exportType,
      requestParams
  );
  writeBinaryResponse(response, result.data(), result.contentType(), result.filename());
  return;
}
```

**No existing code is modified** — this is purely additive (new method + new branch).

### Phase C: Report Spec Type in DB (~10 lines, modify existing)

**Option 1 (Recommended):** Reuse `SpecType = "P"` (process) with a flag `IsReport = true` on `SFSpec`.

Reports ARE processes in Etendo (`AD_Process` with `IsReport='Y'`). Adding a new column to `ETGO_SF_SPEC`:

| Column | Type | Default | Purpose |
|--------|------|---------|---------|
| `IsReport` | `CHAR(1)` | `'N'` | Distinguishes report specs from process specs |

**Option 2:** Use a new `SpecType = "R"`. Simpler code but requires updating all existing code that checks `SpecType`.

**Recommendation:** Option 1. A report is a process with extra behavior. Keeps existing process routing intact.

2. Add `addReportPaths()` method:
```java
private void addReportPaths(OpenAPI openAPI, SFSpec spec) {
  String specName = spec.getName();

  // GET /sws/neo/{specName} — describe report (parameters, formats)
  // Same as process describe, plus supportedFormats
  String describePath = BASE_PATH + specName;
  PathItem describeItem = getOrCreatePathItem(openAPI, describePath);
  Operation describeOp = createOperation(
      "Describe " + specName,
      "Returns report metadata: parameters and supported export formats");
  describeOp.responses(new ApiResponses()
      .addApiResponse("200", createJsonResponse("Report metadata", createReportDescribeSchema()))
      .addApiResponse("401", new ApiResponse().description("Unauthorized")));
  describeItem.get(describeOp);

  // POST /sws/neo/{specName} — execute report (returns binary file)
  Operation execOp = createOperation(
      "Generate " + specName,
      "Executes the report and returns the file in the requested format");
  execOp.setRequestBody(createReportRequestBody(spec));
  execOp.responses(new ApiResponses()
      .addApiResponse("200", createBinaryResponse("Report file (PDF, XLS, HTML, CSV)"))
      .addApiResponse("400", createJsonResponse("Invalid parameters", null))
      .addApiResponse("401", new ApiResponse().description("Unauthorized")));
  describeItem.post(execOp);

  openAPI.getPaths().addPathItem(describePath, describeItem);
}

private ApiResponse createBinaryResponse(String description) {
  ApiResponse response = new ApiResponse().description(description);
  Content content = new Content();
  // PDF as default media type
  content.addMediaType("application/pdf",
      new MediaType().schema(new Schema<byte[]>().type("string").format("binary")));
  content.addMediaType("application/vnd.openxmlformats-officedocument.spreadsheetml.sheet",
      new MediaType().schema(new Schema<byte[]>().type("string").format("binary")));
  content.addMediaType("text/html",
      new MediaType().schema(new Schema<byte[]>().type("string").format("binary")));
  content.addMediaType("text/csv",
      new MediaType().schema(new Schema<byte[]>().type("string").format("binary")));
  response.setContent(content);
  return response;
}
```

**Request body schema for reports:**

  "params": {
    "dateFrom": "2026-01-01",
    "dateTo": "2026-03-12",
    "orgId": "..."

## URL Design

| Method | Path | Purpose | Response |
|--------|------|---------|----------|
| `GET` | `/sws/neo/{specName}` | Describe report (params + formats) | JSON |
| `POST` | `/sws/neo/{specName}` | Execute report | Binary file |

**Same URL pattern as processes.** The `IsReport` flag on SFSpec determines whether the POST returns JSON (process result) or binary (report file).

**Request:**
```json
POST /sws/neo/general-ledger-report
Authorization: Bearer <jwt>
Content-Type: application/json

{
  "exportType": "pdf",
  "params": {
    "dateFrom": "2026-01-01",
    "dateTo": "2026-03-12"
  }
}
```

**Response:**
```
HTTP/1.1 200 OK
Content-Type: application/pdf
Content-Disposition: attachment; filename="general-ledger-report-2026-03-12.pdf"
Content-Length: 45231

<binary PDF data>
```

## Testing Strategy

### Contract Tests (Schema Forge, Node.js)
- Report contract has `type: "report"` and `supportedFormats`
- Parameters extracted correctly from `AD_Process_Para`
- API prediction includes binary response note

### Unit Tests (JUnit, com.etendoerp.go)
- `NeoReportService.resolveTemplatePath()` — correct path resolution
- `NeoReportService.buildJasperParams()` — parameter type mapping
- `NeoReportService.buildFilename()` — filename generation with date
- Export type parsing (pdf → PDF, xls → XLS, etc.)

### Integration Tests (JUnit, OBBaseTest)
- Execute a known report (e.g., General Ledger) via NEO endpoint
- Verify binary response with correct Content-Type
- Verify PDF is valid (check magic bytes `%PDF-`)
- Test with invalid export type → 400 error
- Test with missing required params → 400 error
- Test unauthorized access → 401 error

### OpenAPI Verification
- Report specs appear in OpenAPI output (`?tag=EtendoGo`)
- Binary response schema present
- Request body schema includes `exportType` and `params`

## Risk Assessment

| Risk | Mitigation |
|------|------------|
| `ReportingUtils` API changes in future Etendo versions | Pin to current method signatures; add integration test that compiles against them |
| Large reports cause memory issues (byte array in memory) | Phase 2: stream directly to `OutputStream` instead of `ByteArrayOutputStream` |
| Jasper compilation slow on first run | Expected behavior; compiled `.jasper` is cached by Jasper engine |
| Template path resolution differs between dev/prod | Copy resolution logic from `BaseReportActionHandler.getReportTemplatePath()` |
| Session-dependent reports (need OBContext) | NEO already sets OBContext from JWT; should work transparently |
