# Plan: Entity Naming Refactor — tabName instead of tableName

**Status:** Pending
**Branch:** feature/ETP-3546
**Date:** 2026-03-20

## Problem

Entity names are derived from the **database table name** (`toCamelCase(tableName)`):
- `C_Order` → `cOrder` → (simplified) `order`
- `C_OrderLine` → `cOrderLine` → `orderLine`

This causes two problems:

1. **Collision risk:** Two tabs pointing to the same table (e.g., "Header" and "Related Orders" both on `C_Order`) produce the same entity name. This hasn't broken yet only because no window currently has duplicate-table tabs — but it will.

2. **Name mismatch with NEO Headless:** NEO URLs use tab-derived names (`/purchase-order/header`), but the pipeline generates table-derived names (`/purchase-order/order`). The `autoSimplifyEntityName()` function is a patch that strips OBDal prefixes (`c`, `m`, `ad`) but doesn't solve the fundamental issue.

## Solution

Change entity `name` to derive from `tabName` (the AD_Tab name) instead of `tableName`:
- `Header` → `header`
- `Lines` → `lines`
- `Line Tax` → `lineTax`
- `Basic Discounts` → `basicDiscounts`
- `Payment Plan` → `paymentPlan`

This naturally matches NEO Headless entity names and is collision-free (AD_Tab names are unique within a window).

## Impact Assessment

### Artifacts to migrate (small scope)
- **3 decisions.json** files (purchase-order, purchase-invoice, sales-order) — entity keys must be remapped
- **29 processes.json** files — `entity` field must be remapped
- **~76 schema-raw.json** files — will be regenerated by re-extraction (or updated by migration script)
- **contract.json** files — regenerated by pipeline, no manual migration

### Code changes (7 files in cli/src, 0 in tools/app-shell)

The frontend components (`useEntity`, `DetailView`, etc.) receive entity names as props — they don't hardcode names. Only the generators need to change.

## Phases

### Phase 1: Core extractor change
**File:** `cli/src/extract-fields.js`

**Change lines 396 and 400:**
```javascript
// BEFORE
primaryEntity = toCamelCase(tab.tableName);
name: toCamelCase(tab.tableName),

// AFTER
primaryEntity = toCamelCase(tab.tabName);
name: toCamelCase(tab.tabName),
```

`toCamelCase` already handles multi-word names: `"Line Tax"` → `"lineTax"`, `"Basic Discounts"` → `"basicDiscounts"`.

**Line 390 (entityClassname) stays unchanged** — it's the OBDal Java class, not the entity name.

**Verify:** `tabName` is always populated in the SQL query (it comes from `t.Name AS tab_name`). Check for nulls in the dataset — if any tab has no name, fall back to tableName.

### Phase 2: Remove autoSimplifyEntityName

**File:** `cli/src/resolve-curated.js`

Line 349 currently does:
```javascript
const simplifiedName = entityDecision.name || autoSimplifyEntityName(rawEntityName);
```

After Phase 1, raw entity names are already tab-based (`header`, `lines`). The simplification is no longer needed:
```javascript
const simplifiedName = entityDecision.name || rawEntityName;
```

**Keep the function exported** but mark it deprecated — `migrate-to-decisions.js` still references it for legacy migration of old curated files. It becomes a no-op for new extractions.

**File:** `cli/src/generate-contract.js`

Lines 236-238 (the fix we just applied) used `autoSimplifyEntityName` to map process entity names. After Phase 3, process entities will already use tab names. Simplify to direct lookup:
```javascript
const curatedEntity = curatedEntityNames.has(p.entity)
  ? p.entity
  : p.entity; // fallback: use as-is (tab-based names match directly)
```

Or even simpler — just use `p.entity` directly, with a warning if not found in `curatedEntityNames`.

### Phase 3: Migration script for existing artifacts

Create `cli/src/migrate-entity-names.js` — a one-time migration that:

1. **Reads `schema-raw.json`** for each artifact directory
2. **Builds a mapping** of old names → new names using the `tabName` field already present in each entity:
   ```
   cOrder → header
   cOrderLine → lines
   cOrderLineTax → lineTax
   mReservationStock → prereservedQty
   cOrderDiscount → basicDiscounts
   cOrderTax → tax
   finPaymentSchedOrdV → paymentPlan
   finPaymentDetailV → paymentDetails
   ```
3. **Updates `schema-raw.json`:** Replace each entity's `name` and `primaryEntity`
4. **Updates `decisions.json`** (3 files): Remap entity keys in `entities` object
5. **Updates `processes.json`** (29 files): Remap `entity` field in each process
6. **Does NOT touch contract.json or generated frontend** — those are regenerated by the pipeline

**Mapping logic:**
```javascript
function buildNameMap(schemaRaw) {
  const map = {};
  for (const entity of schemaRaw.entities) {
    const oldName = entity.name; // toCamelCase(tableName)
    const newName = toCamelCase(entity.tabName); // toCamelCase(tabName)
    map[oldName] = newName;
  }
  return map;
}
```

**Safety:** The script should:
- Print the full mapping before applying (dry-run by default, `--apply` to execute)
- Detect collisions (two old names mapping to the same new name) and abort
- Back up modified files as `.bak` (or rely on git)

### Phase 4: Re-run pipeline for active windows

After migration, regenerate contracts and frontend for the 3 windows with decisions:
```bash
node cli/src/pipeline.js --menu-name "Purchase Order" --steps resolve-curated,generate-contract,generate-frontend
node cli/src/pipeline.js --menu-name "Purchase Invoice" --steps resolve-curated,generate-contract,generate-frontend
node cli/src/pipeline.js --menu-name "Sales Order" --steps resolve-curated,generate-contract,generate-frontend
```

For the remaining 73 artifacts without decisions.json, they only have schema-raw.json — the migration script updates those, and the pipeline will use the new names when they're eventually classified.

### Phase 5: Verify and clean up

1. **Run existing tests:** `make test` in Schema Forge (contract tests, validation tests)
2. **Spot-check NEO alignment:** Compare entity names in `contract.json` with `ETGO_SF_ENTITY.name` in DB — they should now match naturally without needing `autoSimplifyEntityName`
3. **Verify frontend:** `make dev` and check that Purchase Order loads, shows process buttons, selectors work
4. **Remove `autoSimplifyEntityName` calls** from:
   - `resolve-curated.js` line 349 (replaced in Phase 2)
   - `generate-contract.js` lines 236-238 (simplified in Phase 2)
   - Keep the function itself as deprecated for `migrate-to-decisions.js` backward compatibility
5. **Update `generate-mock-data.js`** line 228: The parent ID field derivation `${primaryEntity}Id` changes from `orderId` to `headerId`. Verify this matches actual API responses.

## Risks

| Risk | Mitigation |
|------|------------|
| Tab names not unique within a window | AD_Tab names are enforced unique per window by Etendo — not a risk |
| Tab names contain special characters | `toCamelCase` already handles spaces, hyphens, etc. |
| Existing NEO DB config uses old entity names | `push-to-neo` re-pushes with new names — entities are matched by tabId, not name |
| `generate-mock-data` parentIdField breaks | Verify `headerId` vs `orderId` in mock data — may need to derive from tabId or relationship metadata instead of entity name |
| Other windows have different tab naming patterns | Run migration with `--dry-run` first to review all mappings |

## Rollback

Git revert the commit + re-run migration with the inverse mapping. All generated files are regenerated from raw + decisions, so only raw artifacts need reverting.
