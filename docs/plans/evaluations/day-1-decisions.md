## D-01 — Vertical Slice Scope: Order + Lines + completeOrder

### What was decided

The vertical slice covers three entities together: the Sales Order header, the Order Lines (master-detail relationship), and the `completeOrder` process. The slice is end-to-end — extraction, decisions, generation, and testing for all three in a single pass.

### Why

A slice with only the header would be technically unrepresentative. A real Sales Order cannot exist without lines; the document is meaningless without them. The `completeOrder` process is the most critical business operation: it orchestrates inventory reservation, tax calculation, accounting posting, and status update in a single OBDal transaction. Including it from day one forces the pipeline to prove it can handle multi-entity relationships and stateful processes — the hardest parts of the problem.

Without `completeOrder`, the slice would validate extraction and generation of fields but not the most important architectural claim: that processes generated by Schema Forge run correctly inside Etendo's transaction model.

### Alternatives considered

| Alternative | Rejected reason |
|-------------|-----------------|
| Header-only slice | Too narrow. Doesn't test master-detail. Doesn't prove process generation. |
| Header + Lines, no process | Defers the hardest problem. Process generation is the highest-risk component. |
| Full Sales Order with all 3 MVP processes (voidOrder, openPeriod) | Too wide for day-1. Risk of overextension before pipeline is validated. |
| Arbitrary window (e.g., Business Partner) | Less representative. BP has no complex processes or master-detail tabs that mirror the target use case. |

### Who decided

Project lead (Etendo Software), captured in Vertical Slice Design document, status: Approved.

### Impact

- Developers working on extraction must handle multi-tab schemas from day one.
- Backend generator must produce event handlers that correctly derive system fields for both the header and lines.
- Contract generator must produce tests for both the parent endpoint and the nested lines endpoint.
- The `completeOrder` implementation validates the single-transaction claim across the entire pipeline.
- Sets the bar: if this slice works end-to-end, the approach is proven.

## D-02 — Approach: Pipeline-First

### What was decided

The team builds components in the exact order they appear in the pipeline: extraction first, then schemas and validators, then pre-classification, then the decision panel, then contract generation, then backend generation, then UI generation. Each phase produces artifacts that the next phase consumes.

### Why

Pipeline-first guarantees that no component is built before its inputs are defined. If you build the contract generator before the schema validator, you are making assumptions about what a valid curated schema looks like. Those assumptions may be wrong. Pipeline-first forces each team to receive real artifacts before they produce real artifacts. It also creates a natural integration test at each boundary: if phase N cannot consume the output of phase N-1, the failure is immediate and localized.

### Alternatives considered

**Outside-in:** Start with the Decision Editor UI, define the UX first, then work inward toward extraction. Rejected because the UI depends on knowing the shape of schema data, which requires running extraction on a real Etendo instance first. Building the UI first would mean building it against invented data, then adjusting it when real data arrives.

**Contract-first:** Define all JSON schemas (schema-raw, schema-curated, rules, processes, contract) before writing any code, then build all components in parallel against those contracts. Rejected for the vertical slice because the schemas themselves are hypotheses until validated against real Etendo data. The field extractor may discover fields or reference types not anticipated at schema definition time. Locking the schema before extraction inverts the learning flow.

**Feature-first:** Pick one user-facing feature (e.g., the Decision Editor) and build it fully before moving to the next. Rejected because it creates integration gaps — a fully-built Decision Editor is useless without a working extractor feeding it.

### Who decided

Project lead, captured in Vertical Slice Design, section 2: Approach: Pipeline-First.

### Impact

- The 4-developer parallelization plan is structured around pipeline waves, not independent features.
- Developers on later phases (F7 backend, F8 UI) cannot start until earlier phases deliver their artifacts — this is intentional.
- Phase boundaries act as natural handoff points for code review.
- Testing strategy follows the same order: contract tests (Node.js) run before compilation; integration tests run after.

## D-03 — Data Source: Real Etendo Instance

### What was decided

The Field Extractor and Rule Extractor connect to a real, running Etendo instance (PostgreSQL database + Java source) from day one. The vertical slice does not use synthetic or mocked schema data.

### Why

The entire value proposition of Schema Forge depends on the extractors capturing the full complexity of a real Etendo window. A Sales Order in production Etendo has approximately 60 fields, many of which feed accounting, inventory, costing, and audit subsystems. These fields are invisible to the user but critical to the document lifecycle. Synthetic data would underrepresent this complexity and lead the team to design a pipeline that only works on simplified inputs.

Furthermore, Etendo's field classification logic (IsDisplayed, IsReadOnly, DefaultValue patterns, AD_Callout registrations, Event Handler registrations) is only fully visible in a live instance. The pre-classifier must handle all the edge cases that real data presents. If the team delays using real data, those edge cases surface late, when fixing them is expensive.

### Alternatives considered

| Alternative | Rejected reason |
|-------------|-----------------|
| Hand-crafted fixture JSON (10-15 fields) | Too simple. Pipeline designed for a toy would need rework for real data. |
| Snapshot of a previous Etendo export | Stale. May miss callouts or handlers added in recent versions. |
| Mock Etendo database (populated by scripts) | Labor-intensive to maintain realistic complexity. Risk of missing edge cases. |

### Who decided

Project lead, captured in Vertical Slice Design header metadata: "Data Source: Real Etendo instance."

### Impact

- Development environment requires PostgreSQL access to a running Etendo instance.
- The DB driver decision (pg / PostgreSQL) is a direct consequence of this choice.
- The pre-classifier must be robust to real-world naming patterns, not just clean test data.
- Integration tests (JUnit, OBBaseTest) run on the same real instance, giving a single source of truth for test data.

## D-04 — Decision Tools: Web UI Minima (Single Panel)

### What was decided

The Decision Editor and Rule Catalog, which in the PRD are described as separate web UI tools, are merged into a single `decision-panel` React application for the vertical slice. The panel provides the minimum viable interface: field visibility buttons (editable / readOnly / system / discard) and rule decision buttons (Keep / Replace / Simplify / Omit).

### Why

The PRD defines the Decision Editor and Rule Catalog as full standalone tools. For the vertical slice, building two separate tools doubles the frontend development effort without adding demonstrative value. The purpose of the slice is to prove the pipeline end-to-end, not to deliver production-quality tooling UX. A single panel that lets a human make all required decisions is sufficient to produce the curated artifacts the rest of the pipeline needs.

The PRD explicitly acknowledges this tradeoff: the Process Designer is listed as "MVP: JSON" (manual editing), and the Permission Matrix as "manual XML." Extending that simplification to the decision tools for the vertical slice is consistent.

### Alternatives considered

| Alternative | Rejected reason |
|-------------|-----------------|
| Full separate Decision Editor + Rule Catalog | Double the frontend work for no additional pipeline validation. |
| CLI-only decisions (edit JSON in a text editor) | No visual impact display. Pre-classifier recommendations invisible to the human. Misses the key UX requirement: the human sees IA recommendations before deciding. |
| No decision UI at all (edit curated JSON directly) | Same problem as CLI-only, plus error-prone JSON editing on ~60 fields. |

### Who decided

Project lead, Vertical Slice Design section 3.1: "Simplifications for Vertical Slice."

### Impact

- The `tools/decision-panel/` directory replaces the separate `tools/decision-editor/` and `tools/rule-catalog/` from the TDD structure.
- Frontend developers build one app instead of two.
- The UX is minimal: the slice does not validate the full Decision Editor product experience.
- Post-slice work will split this panel back into separate tools with full production UX.

## D-05 — Repository: Monorepo with npm Workspaces

### What was decided

All packages (CLI tools, web tools, schemas, templates, core-maps) live in a single repository managed with npm workspaces. There is no separate package manager (Turborepo, Nx, Lerna) on top.

### Why

Schema Forge has multiple packages that share types and utilities: the CLI tools, the React decision panel, the UI generator, and the JSON schemas. A monorepo keeps these co-located so a developer can make a change to a shared schema definition and immediately see the effect on both the CLI and the web tools without publishing packages or managing version pins.

npm workspaces (built into npm 7+) provide workspace hoisting and cross-package script running without any additional tooling. For the vertical slice, this is sufficient. More complex build orchestration (Turborepo caching, affected-graph execution) adds setup cost without delivering value until the codebase has many more packages and long build times.

### Alternatives considered

| Alternative | Rejected reason |
|-------------|-----------------|
| Separate repositories per package | Cross-package changes require coordinated PRs. Schema changes require version bumps and publishes. Too much friction for a small team moving fast. |
| Monorepo with Turborepo | Adds caching and dependency graph execution. Valuable at scale, but setup overhead is not justified for a single-window vertical slice. |
| Monorepo with Nx | Similar to Turborepo. More opinionated, steeper learning curve. Not justified for vertical slice size. |
| Single flat package (no workspaces) | Conflates CLI, web tools, and schemas into one package.json. Breaks isolation between concerns. |

### Who decided

Project lead, Vertical Slice Design section 3 (Repository Structure shows root package.json with workspace reference) and section 4 (Technology Decisions: "No complex monorepo manager — npm workspaces basic").

### Impact

- Root `package.json` defines workspaces pointing to `cli/`, `tools/decision-panel/`, `tools/ui-generator/`.
- Scripts in each workspace are runnable from root via `npm run <script> --workspace=cli`.
- Shared code (type definitions, validation utilities) can be extracted to a `packages/shared/` workspace when needed.
- No build cache: every `npm run build` is a full build. Acceptable for slice size.

## D-06 — Template Engine: Handlebars

### What was decided

All code generation templates (Java EventHandler, DalProcess, RxEndpoint, DTO, XML dataset) use Handlebars (`.hbs` files). The backend generator loads these templates and renders them with data from the curated artifacts.

### Why

Code generation requires a template engine that separates template structure from generation logic. The generated files are Java and XML — formats with significant boilerplate that must be reproduced exactly. Handlebars is logic-less by design: it does not allow arbitrary JavaScript in templates. This is a feature, not a limitation. Logic-less templates force all generation logic into the generator script, making templates readable by anyone who knows Java or XML (not just JavaScript developers) and making the generation logic testable in isolation.

Handlebars is well-known, has stable Node.js support, handles partials (reusable template fragments) well, and produces readable output with predictable whitespace behavior. It is explicitly listed in the TDD as the chosen template engine.

### Alternatives considered

| Alternative | Rejected reason |
|-------------|-----------------|
| EJS (Embedded JavaScript) | Allows arbitrary JS in templates. Logic bleeds into templates, making them hard to read and test. |
| Mustache | Handlebars is a superset of Mustache with helpers and partials. Mustache lacks helpers needed for conditional block generation. |
| Template literals (JavaScript string interpolation) | No file-based template system. Templates become JavaScript strings in the generator. Hard to read, hard to maintain, no syntax highlighting. |
| Custom DSL | Maximum control, maximum maintenance burden. Unjustified for the scope. |
| jte (Java Template Engine) | Would require a Java process to run templates. Breaks the CLI-is-Node.js constraint. |

### Who decided

Project lead, Vertical Slice Design section 4 (Technology Decisions): "Template engine: Handlebars — Simple, logic-less, well-known."

### Impact

- Template files live in `templates/` with `.hbs` extension.
- The backend generator (`cli/src/generate-backend.js`) is the only code that calls Handlebars.
- Adding a new generated file type requires a new `.hbs` template and a corresponding render call in the generator.
- Helpers (e.g., `toJavaType`, `toUpperCamel`) are registered in the generator and reusable across all templates.

## D-07 — Process Designer: Manual JSON Editing

### What was decided

Process definitions (`processes.json`) are authored by hand as JSON files. There is no web UI for the Process Designer in the vertical slice.

### Why

The PRD explicitly scopes the Process Designer interactive UI as post-MVP: "Process Designer interactive UI (MVP: JSON)." A web UI for process design is a significant frontend development effort. The JSON schema for processes is well-defined (preconditions, steps, behavioral contracts, edge cases, rollback declarations). A developer or consultant familiar with the schema can author a correct `processes.json` directly.

For the vertical slice, there is exactly one process to define: `completeOrder`. The cost of building a web UI to define one process outweighs the benefit. The contract validator (`validate-processes.js`) provides immediate feedback if the manually authored JSON violates the schema or missing required elements (e.g., fewer than 3 edge cases).

### Alternatives considered

| Alternative | Rejected reason |
|-------------|-----------------|
| Full Process Designer web UI | High frontend effort. Not justified for 1-3 processes in the vertical slice. |
| Conversational AI process definition | Interesting idea. Requires prompt engineering and output parsing. Deferred: higher risk, not needed to prove the pipeline. |
| YAML instead of JSON | YAML is more human-readable for process definitions. Rejected for consistency — all other artifacts (schema-raw, schema-curated, rules, contract) are JSON. Adding YAML requires a parser and a transformation step. |

### Who decided

Project lead, PRD section 14.2 (Out of Scope): "Process Designer interactive UI (MVP: JSON)." Reinforced in Vertical Slice Design section 3.1.

### Impact

- `artifacts/sales-order/processes.json` is hand-authored by the team.
- `cli/src/validate-processes.js` validates the file against `schemas/processes.schema.json` and enforces the edge-case minimum.
- No tooling UI for process design in the vertical slice deliverable.
- The process definitions for `voidOrder` and `openPeriod` (full MVP) will also be manual JSON.

## D-08 — Evaluation Pipeline: 3 Evaluators + Sage Before Coding

### What was decided

Before any code is written, four parallel evaluations are run on the design documents:

1. **Product Owner** — reviews scope against PRD, checks priority alignment, assesses success criteria achievability.
2. **Architect** — reviews tech stack viability, repository structure, dependency risks, identifies architectural blind spots.
3. **External Advisor** — reviews risks from similar projects, identifies questionable assumptions, common traps in ERP generation tools.
4. **Sage (Documentation)** — captures day-1 decisions and produces the onboarding guide (this document and its companion).

Each evaluator produces a report. All reports are synthesized into a consolidated Risk Register before development starts.

### Why

Schema Forge is architected at the intersection of ERP metadata extraction, AI-driven classification, multi-language code generation, and real-time UI tooling. Each domain has its own failure modes. No single reviewer covers all of them. Running evaluations in parallel before coding starts ensures that:

- Fundamental architectural risks are identified before they are encoded in code.
- The team starts with a shared, written understanding of the design.
- Day-1 decisions are captured while context is fresh, creating a record for future team members.
- The Risk Register serves as a living document that guides development priorities.

The cost of the evaluation pipeline is one day. The cost of discovering a fundamental design flaw after 3 weeks of coding is significantly higher.

### Alternatives considered

| Alternative | Rejected reason |
|-------------|-----------------|
| No formal evaluation — just start coding | Fast in the short term. High probability of rework when assumptions fail. |
| Single reviewer evaluates all dimensions | One person cannot hold expertise across product, architecture, risk, and documentation simultaneously. |
| Sequential reviews (PO first, then Architect, then...) | Slower. Evaluators cannot benefit from each other's findings. Parallel is as rigorous and twice as fast. |
| Evaluation after first sprint | Defers risk identification. Some architectural decisions (monorepo structure, template engine) are expensive to change after code exists. |

### Who decided

Project lead, Vertical Slice Design section 5 (Day-1 Evaluation Pipeline).

### Impact

- This document is a direct output of D-08.
- The evaluations directory (`docs/plans/evaluations/`) holds all four reports.
- Development does not start until the Risk Register is complete.
- The evaluation process sets the norm: future design changes of comparable scope should go through a similar (possibly lighter) review before implementation.

## D-09 — Team Structure: 4 Exploratory Devs + Balanced Reviewer + Methodical QA + Comprehensive Docs

### What was decided

The team working on Schema Forge operates with four distinct roles:

| Role | Profile | Responsibility |
|------|---------|----------------|
| Exploratory Developers (x4) | Fast movers, comfortable with ambiguity | Implement pipeline phases in parallel waves |
| Balanced Reviewer (x1) | Strong in both product judgment and technical rigor | Reviews PRs for correctness, scope alignment, and test coverage |
| Methodical QA (x1) | Systematic, edge-case focused | Validates contract tests, integration tests, and process edge cases |
| Comprehensive Docs (x1 — Sage) | Big-picture, thorough | Captures decisions, produces onboarding, maintains architectural record |

Developers work in worktree isolation. Each feature is developed in a separate git worktree, reviewed by the Balanced Reviewer, validated by QA, and documented by Sage.

### Why

Schema Forge builds several components that are highly interdependent but independently implementable within a pipeline wave. Four developers working in parallel across F1a (extract-fields), F1b (extract-rules), F5 (process definitions), and core-maps in Wave 1 maximizes throughput without coordination overhead, because each developer owns a fully independent deliverable with a well-defined output artifact.

The Reviewer role is critical because the pipeline has tight interface contracts: if the field extractor outputs a schema-raw.json that doesn't match what the validator expects, every downstream component fails. A single reviewer who understands both the product intent and the technical contracts catches mismatches before they propagate.

The QA role focuses on edge cases because Schema Forge's correctness depends on it: a system field that nobody fills, a process that doesn't roll back on failure, or a contract test with wrong type expectations are all silent failures — the pipeline looks green but the generated module is wrong.

### Alternatives considered

| Alternative | Rejected reason |
|-------------|-----------------|
| All developers are generalists, no specialization | Roles naturally emerge; making them explicit reduces ambiguity and sets clear expectations for each deliverable. |
| QA embedded within each developer's work | Developers test their own code less rigorously. Independent QA finds what the author missed. |
| No documentation role | Day-1 decisions are not recorded. Onboarding new team members requires lengthy 1:1 sessions. Knowledge is tacit and non-transferable. |
| One developer per pipeline phase (sequential) | Wave-based parallelism is the explicit design. Sequential execution multiplies the timeline by 4. |

### Who decided

Implicit in the Vertical Slice Design parallelization table (section 2.3) and the evaluation pipeline (section 5). Role definitions emerged from the evaluation pipeline structure. Captured explicitly here for the first time.

### Impact

- Developers use git worktrees per feature — no long-lived feature branches accumulate drift.
- Code review is not optional: every phase output goes through the Reviewer before feeding downstream phases.
- QA owns the contract test suite and the integration test suite as a first-class deliverable, not an afterthought.
- Sage produces two documents per design iteration: a decisions record and an onboarding guide.
