# Form Migration Guide

## Overview

AD_Forms are custom UI components with no extractable metadata. Unlike Windows (AD_Tab + AD_Column) and Processes (AD_Process_Para), forms are pure Java + HTML implementations. Migration is artisanal — each form requires manual analysis and custom implementation.

## When You'll See This

When using `--menu-id` or `--menu-name` with a menu entry that has `action = 'X'` (Form), the pipeline cannot auto-generate code. Instead, follow this guide.

## Step 1: Analyze the Legacy Form

- Find the Java class: `src/org/openbravo/erpCommon/ad_forms/{ClassName}.java`
- Find the HTML template: same path, `.html` extension
- Identify:
  - Input fields (name, type, label)
  - Actions/buttons
  - Grid columns (if any)
  - SQL queries in the Java class

## Step 2: Categorize the Form

| Category | Examples | Migration Path |
|----------|----------|----------------|
| Batch Process | Create Invoices from Orders, Create Shipments | Filter -> Grid -> Select -> Execute pattern |
| Admin/Config | Initial Client Setup, Alert Management | Internal Etendo — usually skip |
| System UI | Menu, Logout, Session Preferences | Infrastructure — skip |
| Dev Tools | SQL Query, HQL Query, Audit Trail | Developer tools — skip |
| Business | Pending Goods Receipts, GL Posting | Case by case |

## Step 3: Create the NeoHandler (Backend)

Create a Java class implementing `NeoHandler` with `@Named("your-qualifier")`:
- GET -> query/filter logic (what the "Search" button did)
- POST -> execution logic (what the "Process" button did)
- Register in ETGO_SF_ENTITY with `javaQualifier` matching `@Named` value

Example:

```java
@Named("generate-invoices")
public class GenerateInvoicesHandler implements NeoHandler {
  @Override
  public NeoResponse handle(NeoContext ctx) {
    if ("GET".equals(ctx.getHttpMethod())) {
      // Query pending orders based on filter params
      return NeoResponse.ok(queryPendingOrders(ctx.getQueryParams()));
    }
    if ("POST".equals(ctx.getHttpMethod())) {
      // Execute invoice generation for selected order IDs
      return NeoResponse.ok(generateInvoices(ctx.getRequestBody()));
    }
    return null;
  }
}
```

## Step 4: Register in NEO Headless

Use `push-to-neo.js` or direct DB writes:
- `ETGO_SF_SPEC`: specName, specType="P"
- `ETGO_SF_ENTITY`: entityName="main", javaQualifier="your-qualifier"
- `ETGO_SF_FIELD`: one per parameter/filter field

## Step 5: Create the React Component

Create a custom React component — NOT generated by template. Place it in:
`artifacts/{form-name}/generated/web/{form-name}/index.jsx`

The component receives `token`, `apiBaseUrl`, `windowName`, `recordId` from WindowLoader.
Use the same UI components as other windows (inputs, tables, buttons from the app-shell).

Typical pattern for batch-process forms:
1. Filter inputs (date range, business partner, org)
2. Search button -> GET to NeoHandler -> populate grid
3. Selectable grid with results
4. Execute button -> POST selected IDs to NeoHandler
5. Result display

## Step 6: Register in the App Shell

1. Add entry to `tools/app-shell/src/windows/registry.js`:
   ```js
   'generate-invoices': () => import('@generated/generate-invoices/generated/web/generate-invoices/index.jsx'),
   ```
2. Add entry to `tools/app-shell/src/menu.json` in the appropriate group
