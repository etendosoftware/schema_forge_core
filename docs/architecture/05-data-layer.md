# 05 — Data Layer

Database architecture and data access patterns for generated Etendo modules in production: OBDal internals, transaction model, connection management, data integrity, and backup/recovery.

---

## 1. OBDal Architecture

### 1.1 What OBDal Is

OBDal is Etendo's data access layer — a wrapper around Hibernate ORM that provides:

- Simplified CRUD: `save()`, `remove()`, `get()`, `createQuery()`
- Automatic context injection (organization, client, user filtering)
- Transaction management: `flush()`, `commitAndClose()`, `rollbackAndClose()`
- Audit column management (created, createdBy, updated, updatedBy)

Generated modules **never interact with JDBC directly**. All database operations go through OBDal, which delegates to Hibernate, which executes SQL against PostgreSQL.

### 1.2 Hibernate Configuration

Etendo uses **XML-based Hibernate mappings** (`hbm.xml`), not JPA annotations. Each entity (table) has a corresponding mapping file generated by Etendo's build process:

```
modules/com.etendoerp.go.{window}/
  src-db/
    database/
      model/
        tables/
          {TABLE_NAME}.xml       ← Etendo table definition
  src/
    com/etendoerp/go/{window}/
      entity/
        {Entity}.java            ← generated entity class
        {Entity}.hbm.xml         ← Hibernate mapping
```

**Key mapping details:**
- Primary keys are `VARCHAR(32)` columns mapped to `java.lang.String`.
- Foreign keys map to Hibernate `many-to-one` associations (lazy by default).
- All entities extend `BaseOBObject`, which provides audit columns and `isActive`.
- `hbm.xml` files are regenerated during `smartbuild` — never edit them manually.

### 1.3 OBContext

Every database operation runs within an `OBContext` that defines:

| Property | Source | Purpose |
|----------|--------|---------|
| `currentUser` | AD_Session | Audit columns (`createdBy`, `updatedBy`) |
| `currentRole` | AD_Session | Access control (window/process permissions) |
| `currentOrganization` | AD_Session | Data filtering (multi-org isolation) |
| `currentClient` | AD_Session | Top-level tenant isolation |
| `currentWarehouse` | AD_Session | Warehouse context for logistics |
| `language` | AD_Session | Translatable field resolution |

OBContext is set at the beginning of each HTTP request (by the authentication layer) and remains constant for the request's lifetime. Hibernate interceptors read OBContext to inject organization and client filters into every query.

### 1.4 Automatic Audit Columns

Every Etendo entity has five audit columns managed automatically by OBDal:

| Column | Type | Set When | Set To |
|--------|------|----------|--------|
| `created` | `TIMESTAMP` | Insert | Current timestamp |
| `createdBy` | `VARCHAR(32)` | Insert | `OBContext.currentUser` |
| `updated` | `TIMESTAMP` | Insert, Update | Current timestamp |
| `updatedBy` | `VARCHAR(32)` | Insert, Update | `OBContext.currentUser` |
| `isActive` | `CHAR(1)` | Insert | `'Y'` (default) |

Generated modules **must not set these columns explicitly**. OBDal handles them via Hibernate interceptors. Attempting to override them may cause conflicts or be silently ignored.

## 2. Transaction Model

### 2.1 Single Transaction Per Request

Each HTTP request to a RequestHandler operates within a single Hibernate transaction:

```
HTTP Request arrives
  │
  ▼
Transaction begins (automatic, container-managed)
  │
  ├── doPost(): create entity, set fields
  │     ├── OBDal.save(entity)
  │     ├── flush() → triggers EventHandler (beforeSave)
  │     │     ├── EventHandler computes derived fields
  │     │     ├── EventHandler may throw → ROLLBACK entire request
  │     │     └── EventHandler succeeds → derived fields persisted
  │     └── Entity saved in session (not yet committed)
  │
  ▼
Transaction commits (automatic, end of request)
  │
  ▼
HTTP Response sent
```

### 2.2 All-or-Nothing Semantics

If **any** exception occurs during the request lifecycle, the **entire** transaction rolls back:

- Validation failure in the RequestHandler → rollback.
- EventHandler throws an exception during derivation → rollback.
- Database constraint violation (unique, FK, check) → rollback.
- OutOfMemoryError or other runtime exception → rollback.

**No partial writes.** Either all changes from the request are committed, or none are. This eliminates the need for compensation logic, Sagas, or distributed transactions.

### 2.3 Explicit Commit in Processes

`DalProcess` implementations are the exception to container-managed transactions. Processes may need to commit mid-execution (e.g., after processing each batch of records):

```java
// Process pseudocode
for (Order order : ordersToProcess) {
    processOrder(order);
    OBDal.getInstance().flush();
}
OBDal.getInstance().commitAndClose();
```

**Risk:** If the process fails after a partial commit, some records are processed and some are not. Generated processes should either:
- Process everything in a single transaction (preferred for small batches).
- Implement idempotent processing so re-running the process is safe.

### 2.4 Flush vs Commit

| Operation | What It Does | When to Use |
|-----------|-------------|-------------|
| `flush()` | Writes pending changes to PostgreSQL but does NOT commit. Transaction still open. Triggers EventHandlers. | Before reading derived values computed by EventHandlers. |
| `commitAndClose()` | Commits the transaction and closes the Hibernate session. | End of process execution. Never in RequestHandlers (container-managed). |
| `rollbackAndClose()` | Rolls back all changes and closes session. | Error handling in processes. |

## 3. Connection Management

### 3.1 HikariCP Connection Pool

Etendo uses HikariCP as its JDBC connection pool. Configuration lives in `Openbravo.properties`:

```properties
bbdd.url=jdbc:postgresql://localhost:5432/etendo
bbdd.driver=org.postgresql.Driver
bbdd.maxActive=50
bbdd.maxIdle=10
bbdd.minIdle=5
bbdd.maxWait=30000
```

These properties are mapped to HikariCP settings at initialization:

| HikariCP Setting | Typical Production Value | Purpose |
|-----------------|------------------------|---------|
| `maximumPoolSize` | 30–50 | Maximum concurrent connections to PostgreSQL |
| `minimumIdle` | 5–10 | Connections kept ready for immediate use |
| `connectionTimeout` | 30,000 ms | Max wait for a connection from pool before error |
| `idleTimeout` | 600,000 ms (10 min) | Close idle connections to free DB resources |
| `maxLifetime` | 1,800,000 ms (30 min) | Recycle connections to prevent stale state |
| `leakDetectionThreshold` | 60,000 ms (1 min) | Log warning if connection held without return |

### 3.2 Connection Lifecycle

```
Request arrives
  → HikariCP provides connection from pool
    → Hibernate session bound to connection
      → OBDal operations execute SQL
      → Transaction commits or rolls back
    → Hibernate session closed
  → Connection returned to pool
Request completes
```

### 3.3 Pool Sizing Guidelines

**Formula:** `maximumPoolSize = (number of concurrent users × avg queries per request) / avg query duration`

Practical guidance:

| Concurrent Users | Recommended Pool Size | Reasoning |
|-----------------|----------------------|-----------|
| < 20 | 10–15 | Low contention, most connections idle |
| 20–100 | 20–30 | Moderate contention during peak hours |
| 100–500 | 30–50 | High contention; monitor for pool wait events |
| > 500 | 50+ (requires PostgreSQL tuning) | May need PgBouncer or connection multiplexing |

**PostgreSQL limit:** `max_connections` in `postgresql.conf` must be >= HikariCP `maximumPoolSize` + overhead for superuser connections, monitoring, and replication.

### 3.4 Multi-Tenant Data Isolation

Etendo uses **single-database, multi-organization** isolation:

- All tenants share the same PostgreSQL database and tables.
- Every table has `ad_org_id` and `ad_client_id` columns.
- OBContext injects `WHERE ad_org_id IN (...)` into every Hibernate query.
- A user with Role A (org: Spain) cannot see data from org: Germany.

This is **not row-level security in PostgreSQL** — it is application-level filtering via Hibernate interceptors. If an application bypasses OBDal (direct JDBC or raw SQL), the organization filter is NOT applied.

## 4. Schema Considerations

### 4.1 ID Columns Are Strings

All `_ID` columns in Etendo are `VARCHAR(32)`:

```sql
-- Legacy IDs (look numeric, but are VARCHAR)
SELECT * FROM ad_reference WHERE ad_reference_id = '19';    -- String/Text
SELECT * FROM ad_reference WHERE ad_reference_id = '30';    -- List

-- Modern IDs (UUIDs without hyphens)
SELECT * FROM c_order WHERE c_order_id = '95E2A8B50A254B2AAE6774B8C2F28120';
```

**Rule:** Always treat `_ID` columns as strings. Always quote values in SQL. Never cast to integer.

### 4.2 Schema Ownership

The database schema is **owned by Etendo**, not by generated modules:

- Tables are created by Etendo's `smartbuild` process (from `src-db/database/model/` XML definitions).
- Column additions require Etendo's module development workflow (not raw `ALTER TABLE`).
- Indexes are managed by Etendo (some auto-created for PKs and FKs, others declared in model XML).
- Generated modules read/write data through OBDal; they never execute DDL.

### 4.3 Common Column Types

| Etendo Type | PostgreSQL Type | Java Type | Notes |
|-------------|----------------|-----------|-------|
| String | `VARCHAR(n)` | `String` | Most text fields |
| ID | `VARCHAR(32)` | `String` | Primary and foreign keys |
| Integer | `NUMERIC(10,0)` | `Long` | Whole numbers |
| Amount | `NUMERIC` | `BigDecimal` | Financial amounts (precision varies) |
| Quantity | `NUMERIC` | `BigDecimal` | Quantities |
| Date | `TIMESTAMP` | `Date` | All dates include time component |
| Yes/No | `CHAR(1)` | `Boolean` | `'Y'` / `'N'` in DB, mapped to boolean |
| List | `VARCHAR(60)` | `String` | Enum-like, validated against AD_Ref_List |
| TableDir | `VARCHAR(32)` | FK entity | Direct FK reference to another table |
| Table | `VARCHAR(32)` | FK entity | FK via AD_Reference_Value configuration |

## 5. Data Integrity

### 5.1 EventHandler Derivations

EventHandlers are CDI observers on `EntityPersistenceEvent` that fire during `flush()`. In generated modules, they compute **system field derivations** — fields that are automatically calculated, not user-entered:

```java
// Fires before save
public void onSave(@Observes EntityNewEvent event) {
    Entity entity = (Entity) event.getTargetInstance();

    // fromConfig: value from system configuration
    entity.setWarehouse(OBContext.getOBContext().getWarehouse());

    // computed: derived from other fields
    entity.setNetAmount(entity.getGrossAmount().subtract(entity.getTaxAmount()));

    // sequence: auto-incremented document number
    entity.setDocumentNo(SequenceUtil.next("PO"));
}
```

**Derivation types:**
| Type | Source | Example |
|------|--------|---------|
| `fromConfig` | System/org configuration | Default warehouse, default currency |
| `fromParent` | Parent tab record | Header's business partner → line's business partner |
| `fromField` | Another field on same record | documentDate → accountingDate (if not set) |
| `lookup` | Query against another table | Product ID → product name, price |
| `computed` | Calculation from other fields | qty × price = lineAmount |
| `sequence` | Auto-increment sequence | Document number generation |

### 5.2 Database Constraints

PostgreSQL enforces several layers of data integrity below OBDal:

| Constraint Type | Enforced By | Example |
|----------------|-------------|---------|
| Primary key | `PRIMARY KEY` | `c_order_pkey` |
| Foreign key | `REFERENCES` | `c_orderline.c_order_id REFERENCES c_order` |
| Not null | `NOT NULL` | Mandatory fields from AD_Column configuration |
| Unique | `UNIQUE` | Document number within organization |
| Check | `CHECK` | `isActive IN ('Y', 'N')` |

If OBDal attempts to persist data violating a constraint, PostgreSQL rejects the write and Hibernate throws a `ConstraintViolationException`, which rolls back the transaction.

### 5.3 Etendo Database Triggers

Etendo has legacy **database-level triggers** on some core tables. These execute in PostgreSQL, outside of Hibernate's awareness:

- Triggers may modify data after Hibernate writes (Hibernate cache becomes stale).
- Generated modules should not add database triggers — use EventHandlers instead.
- If a generated module writes to a table that has existing Etendo triggers, the trigger executes and may have side effects (e.g., updating related records, enforcing business rules).

**Discovery required:** Before generating a module for a window, check for existing triggers on the target table:

```sql
SELECT trigger_name, event_manipulation, action_statement
FROM information_schema.triggers
WHERE event_object_table = '{table_name}';
```

### 5.4 Optimistic Locking

Etendo uses **optimistic locking** via the `updated` timestamp column:

```
User A loads record (updated = 2026-03-10 14:00:00)
User B loads same record (updated = 2026-03-10 14:00:00)
User A saves → updated = 2026-03-10 14:05:00 → SUCCESS
User B saves → Hibernate checks: expected updated = 14:00:00, actual = 14:05:00
  → StaleObjectStateException → Transaction rolls back
  → User B sees "Record was modified by another user" error
```

This prevents lost updates without database-level locks, but requires the frontend to display a meaningful error and offer the user a way to reload and retry.

### 5.5 Concurrent Access Patterns

| Scenario | Risk | Handling |
|----------|------|----------|
| Two users edit same record | Lost update | Optimistic locking (see above) |
| Two users create records simultaneously | No conflict | Each gets unique UUID primary key |
| Process runs while user edits related record | Stale data in process | Process should reload entities before modifying |
| Bulk import while users work | Lock contention | Schedule imports during low-activity windows |

## 6. Backup and Recovery

### 6.1 PostgreSQL Backup Strategies

| Strategy | Command | Recovery Time | Data Loss Window |
|----------|---------|---------------|------------------|
| **Full dump** | `pg_dump -Fc etendo > backup.dump` | 10–60 min (depends on size) | Since last dump |
| **WAL archiving** | Continuous archiving to remote storage | Minutes (PITR) | Seconds |
| **Base backup + WAL** | `pg_basebackup` + WAL replay | Minutes | Near-zero |
| **Logical replication** | Streaming to standby | Failover in seconds | Near-zero |

**Recommendation for production:**
- **Daily full dump** (`pg_dump`) as baseline.
- **Continuous WAL archiving** for point-in-time recovery (PITR).
- **Pre-deployment snapshot** before every database migration.

### 6.2 Point-in-Time Recovery (PITR)

When a deployment goes wrong (e.g., a database migration corrupts AD records):

```bash
# 1. Stop Etendo
# 2. Stop PostgreSQL
pg_ctl stop -D /var/lib/postgresql/data

# 3. Restore base backup
rm -rf /var/lib/postgresql/data
pg_restore -D /var/lib/postgresql/data base_backup/

# 4. Configure recovery target
cat > /var/lib/postgresql/data/recovery.conf <<EOF
restore_command = 'cp /archive/%f %p'
recovery_target_time = '2026-03-10 13:59:00'   # just before deployment
EOF

# 5. Start PostgreSQL (replays WAL up to target time)
pg_ctl start -D /var/lib/postgresql/data
```

### 6.3 AD Record Registration

AD records (Window, Tab, Field, Process definitions) must be registered in the database for the Etendo Application Dictionary to recognize a generated module. The registration mechanism should be **idempotent on primary key**:

- If the record does not exist → `INSERT`.
- If the record exists with matching PK → `UPDATE`.
- Records not in the migration are **not deleted** (additive only).

**Consequence:** Additive migrations cannot "undo" a previous migration. If version 2 removes a field that version 1 added, the AD_Field record from version 1 remains in PostgreSQL. Explicit cleanup (DELETE) is required.

> **Note:** Database migration strategy (Liquibase) is under evaluation. Not yet implemented. Liquibase changesets would provide structured rollback capabilities to address this limitation.

### 6.4 Environment Migration

Moving data between environments (dev → staging → production):

| What | How | Caveat |
|------|-----|--------|
| Schema structure | `smartbuild` regenerates from model XML | Same Etendo version required |
| AD configuration | Database migration (Liquibase, pending) | Must use same AD IDs across environments |
| Business data | `pg_dump` / `pg_restore` of specific tables | Organization/client IDs must match |
| Test data | Not migrated to production | Use separate dataset for test data |

## 7. Critical Failure Points

### Severity: Red (Data Loss / Service Down)

| Failure | Cause | Detection | Mitigation |
|---------|-------|-----------|------------|
| Connection pool exhaustion | Too many concurrent users; leaked connections from unclosed OBDal sessions | `HikariPool - Connection is not available` in logs; all requests timeout | Monitor pool metrics; set `leakDetectionThreshold`; review code for missing `finally` blocks |
| Transaction deadlock | Two users updating parent/child records in opposite order | PostgreSQL logs deadlock detection; one transaction aborted | Etendo's single-transaction model usually prevents this; if it occurs, retry the failed request |
| EventHandler infinite loop | Derivation A triggers save → fires EventHandler → derivation B triggers save → fires EventHandler | StackOverflowError; Tomcat thread hangs | Guard EventHandlers with `if (alreadyDeriving)` flag; limit recursion depth |
| OBContext misconfiguration | Wrong organization set in context → user sees/modifies data from another org | Data appears from wrong organization; compliance violation | Validate OBContext at RequestHandler entry; never override organization without access check |

### Severity: Yellow (Performance / Data Quality)

| Failure | Cause | Detection | Mitigation |
|---------|-------|-----------|------------|
| Slow queries on large tables | Missing indexes on columns used in HQL WHERE clauses or ORDER BY | Response times > 2s; `pg_stat_statements` shows sequential scans | Analyze query plans (`EXPLAIN ANALYZE`); add indexes to frequently filtered columns |
| Hibernate cache stale | Direct SQL updates (scripts, DB triggers) bypass Hibernate cache | Entity shows old values after reload; inconsistency between API and DB | Call `OBDal.getInstance().getSession().clear()` after known direct DB changes |
| Audit column bypass | Direct SQL update skips EventHandler → `updated`/`updatedBy` not set | Audit trail shows stale timestamps; compliance gap | Never use direct SQL for data modifications in production; always go through OBDal |
| Optimistic lock storms | High-contention record (e.g., sequence counter) causes repeated `StaleObjectStateException` | Users see frequent "modified by another user" errors | Use database sequences for counters; reduce contention surface |
| FK cascade issues | Deleting parent record with child records violates FK constraint | `ConstraintViolationException` on delete | Check for child records before delete; implement cascade soft-delete if needed |
